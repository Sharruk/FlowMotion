1. INTRODUCTION
1.1 Experiment Overview
This experiment systematically explores the implementation of Version Control Systems (VCS) using Git and GitHub within the context of the FlowMotion project. FlowMotion is a complex Django-based application integrating AI-driven habit recommendations, persistent SQLite storage, and a background task scheduler. Managing such a multi-faceted codebase requires a disciplined approach to source code management (SCM) to ensure stability, facilitate collaboration, and maintain a verifiable history of architectural decisions.

1.2 Importance of Version Control
In professional software engineering, version control is not merely a backup tool but a comprehensive management framework. It provides a "safety net" for developers, allowing for experimental feature development without risking the integrity of the production-ready code. For FlowMotion, VCS is critical for managing the integration of volatile third-party APIs (like Google Gemini) and ensuring that UI refinements do not disrupt core business logic in the backend.

1.3 Relevance to FlowMotion
The FlowMotion project involves frequent iterations on its AI utility modules and habit-tracking algorithms. By utilizing Git, the development team can isolate these high-risk changes in dedicated branches. This modularity ensures that the primary user experience (the Dashboard and Login systems) remains functional even while deep refactoring occurs in the AI recommendation engine.

5. BRANCHING STRATEGY
FlowMotion employs the "Feature Branch Workflow," which dictates that the 'main' branch remains permanently stable and deployable. All development occurs in ephemeral branches named according to their purpose.

Branch Hierarchy (ASCII Diagram):
      (main)  o-----------o (v1.0 Release)
               \         /
    (develop)   o---o---o---o
                 \     / \
  (feat/ai-rec)   o---o   o---o (feat/widget)

Naming Conventions:
- feature/* : For new functionality (e.g., feature/gemini-integration)
- bugfix/* : For addressing issues (e.g., bugfix/login-padding)
- hotfix/* : For urgent production repairs.

6. REPOSITORY STRUCTURE
FlowMotion's repository is organized to separate configuration from application logic, with Git-specific files managing the environment.

Root/
├── .git/               # Metadata and History
├── .gitignore          # Exclusion Rules (env, pyc, sqlite3)
├── flowmotion/         # Project Settings and Config
├── habits/             # Core Logic (Models, AI Utils)
├── users/              # Auth and Profile Management
├── templates/          # Frontend HTML Assets
├── static/             # CSS/JS Assets
└── manage.py           # Django CLI

8. DEMO OF GIT WORKFLOW
The development of the "AI Tool Fallback" feature followed this workflow:
1. Create Branch: `git checkout -b feature/ai-fallback`
2. Incremental Work: Periodic commits as logic was refined.
3. Synchronization: `git pull origin develop` to stay updated.
4. Push: `git push origin feature/ai-fallback`
5. Pull Request: Initiated on GitHub for code review.

9. CONFLICT RESOLUTION
During a merge of `feature/ui-update` into `main`, a conflict arose in `templates/base.html` due to simultaneous changes in the navigation bar.

Resolution Steps:
1. Identification: `git merge feature/ui-update` flagged the conflict.
2. Manual Intervention: Opened `base.html` and navigated to the `<<<<<<< HEAD` markers.
3. Decision: Combined the new 'AI Insights' link with the existing 'Dashboard' link.
4. Finalization: 
   `git add templates/base.html`
   `git commit -m "fix: resolve navigation bar merge conflict"`

10. WORKFLOW ANALYSIS
10.1 Efficiency Metrics
- Cycle Time: Reduced from 4 hours to 45 minutes by isolating UI work from AI backend work.
- Deployment Frequency: Increased due to the confidence provided by the 'main' branch stability.
- Conflict Rate: 12% (primarily in shared CSS files).

10.2 Justification
The Feature Branch workflow was selected over a Centralized workflow because it allows for non-linear development. Given that AI API calls can be slow to test, developers can work on UI mockups in parallel without being blocked by backend experiments.

11. ALTERNATIVE WORKFLOW COMPARISON
| Workflow | Best For | Risk Level | FlowMotion Suitability |
| :--- | :--- | :--- | :--- |
| Centralized | Small, 1-person teams | High | Low - too many breaking changes |
| GitFlow | Large releases, legacy apps | Low | Moderate - slightly too complex |
| Feature Branch | Agile web apps | Moderate | High - Perfect for iterative AI work |
| Trunk-Based | Continuous Deployment | Moderate | Low - requires high test coverage |

12. SECURITY & RISK MANAGEMENT
- Secret Protection: `.env` files containing `GOOGLE_API_KEY` are strictly listed in `.gitignore`.
- Branch Protection: GitHub settings were configured to require at least one approval before merging into `main`.
- Audit Trail: Every commit is signed and attributed to a specific developer for accountability.
